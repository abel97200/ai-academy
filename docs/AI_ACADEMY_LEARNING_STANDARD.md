# Standard pédagogique AI Academy

## Statut du document

Ce document définit le standard pédagogique commun à tous les parcours AI Academy.

Il sert de référence pour :

- la conception des parcours ;
- la création des modules et des leçons ;
- la rédaction des exercices et des quiz ;
- la validation des compétences ;
- la conception des projets fil rouge ;
- le travail de ChatGPT, Codex, Claude Code et des futurs contributeurs.

En cas de contradiction entre un contenu local et ce document, la vision produit et les décisions validées du projet restent prioritaires.

---

## 1. Finalité d’AI Academy

AI Academy n’est pas une simple bibliothèque de cours.

Sa finalité est de rendre un débutant capable de :

- comprendre ce qu’il construit ;
- modifier une solution existante ;
- pratiquer avec l’aide de l’IA ;
- tester et vérifier un résultat ;
- expliquer ses choix ;
- publier une application ou un système simple ;
- gagner progressivement en autonomie.

La technologie est un moyen. La compétence démontrée est l’objectif.

---

## 2. Public cible

Le public principal est composé de débutants motivés.

L’apprenant peut :

- avoir peu d’expérience technique ;
- utiliser une IA pour l’aider ;
- ne pas connaître le jargon ;
- manquer de repères pour vérifier un résultat ;
- être capable d’apprendre par petites étapes.

Les contenus doivent rester :

- simples ;
- exacts ;
- progressifs ;
- concrets ;
- peu chargés ;
- orientés vers l’action.

---

## 3. Principes pédagogiques obligatoires

### 3.1 Une notion principale par leçon

Chaque leçon doit viser une notion centrale.

Une leçon ne doit pas devenir un chapitre encyclopédique.

Lorsqu’un sujet est trop large, il doit être découpé.

### 3.2 Pratiquer davantage que lire

L’apprenant doit agir rapidement.

Une leçon ne doit pas se limiter à :

- lire ;
- regarder ;
- répondre à un QCM.

Elle doit conduire à une manipulation, une décision, une production ou une vérification.

### 3.3 Aucun jargon sans définition

Tout terme technique doit être expliqué simplement avant d’être utilisé.

Une analogie concrète peut être utilisée, mais elle ne doit jamais remplacer l’explication exacte.

### 3.4 Le code doit toujours être expliqué

Tout bloc de code doit préciser :

- ce qu’il fait ;
- pourquoi il existe ;
- ce que l’apprenant doit comprendre ;
- ce qu’il doit savoir modifier ;
- ce que l’IA peut produire ;
- ce que l’apprenant doit vérifier lui-même.

### 3.5 La difficulté augmente progressivement

Le parcours doit commencer par des tâches guidées, puis augmenter progressivement :

1. reconnaître ;
2. expliquer ;
3. modifier ;
4. construire ;
5. diagnostiquer ;
6. tester ;
7. sécuriser ;
8. publier ;
9. maintenir.

### 3.6 La simplicité prime

Toujours préférer la solution la plus simple qui atteint l’objectif pédagogique.

Éviter :

- les frameworks prématurés ;
- les architectures lourdes ;
- les abstractions inutiles ;
- les projets trop ambitieux ;
- les fonctionnalités qui n’aident pas l’apprentissage.

---

## 4. Architecture pédagogique d’un module

Chaque module doit former une compétence réelle.

Il doit comporter autant que possible six étapes :

1. **Comprendre** — expliquer le concept et son utilité ;
2. **Observer** — montrer une démonstration ou un exemple annoté ;
3. **Construire** — faire créer ou modifier une première version ;
4. **Provoquer un échec** — introduire une erreur réaliste ;
5. **Corriger et sécuriser** — diagnostiquer, tester et améliorer ;
6. **Intégrer et valider** — ajouter le résultat au projet fil rouge.

Chaque module doit préciser :

- un objectif mesurable ;
- des prérequis ;
- des compétences ;
- un livrable ;
- une évolution du projet fil rouge ;
- un scénario d’échec ;
- une validation pratique ;
- une grille d’évaluation.

---

## 5. Architecture pédagogique d’une leçon

Une leçon doit suivre une structure courte et active.

### 5.1 Brief

Le brief présente :

- la mission ;
- la notion principale ;
- le résultat attendu ;
- l’intérêt concret.

### 5.2 Comprendre

L’explication doit être courte et techniquement exacte.

Elle doit répondre à :

- qu’est-ce que c’est ?
- à quoi cela sert-il ?
- dans quel cas l’utiliser ?
- quelle erreur fréquente éviter ?

### 5.3 Observer

La leçon doit montrer un exemple :

- schéma ;
- démonstration ;
- avant/après ;
- trace d’exécution ;
- extrait de code ;
- comparaison.

### 5.4 Construire

L’apprenant doit réaliser une action concrète :

- modifier un fichier ;
- compléter un code ;
- formuler une instruction ;
- choisir une architecture ;
- lancer une commande ;
- tester un résultat ;
- produire un document.

### 5.5 Vérifier

La leçon doit fournir :

- le résultat attendu ;
- une checklist ;
- les erreurs fréquentes ;
- une correction ;
- un moyen de vérifier que cela fonctionne.

### 5.6 Réfléchir

L’apprenant doit reformuler ou justifier :

- ce qu’il a compris ;
- ce qu’il a modifié ;
- pourquoi la solution fonctionne ;
- ce qu’il contrôlerait dans un autre contexte.

### 5.7 Étendre

Un défi optionnel peut être proposé lorsque cela renforce réellement la compétence.

Il ne doit pas introduire une nouvelle notion majeure.

---

## 6. Missions

Les parcours doivent être organisés autour de missions concrètes.

Une mission correspond à un objectif visible, par exemple :

- rendre un formulaire fonctionnel ;
- connecter un outil ;
- sécuriser une action ;
- corriger une erreur ;
- publier une version ;
- ajouter une capacité au projet fil rouge.

Une mission peut regrouper plusieurs leçons.

Elle doit se terminer par une preuve observable.

---

## 7. Laboratoires

Les laboratoires servent à apprendre par l’erreur.

Ils doivent proposer :

- une situation volontairement incorrecte ;
- un symptôme observable ;
- une hypothèse à formuler ;
- une cause à identifier ;
- une correction à appliquer ;
- un test final.

Exemples :

- sortie JSON invalide ;
- variable d’environnement absente ;
- appel d’outil incorrect ;
- composant qui ne se met pas à jour ;
- boucle infinie ;
- permission excessive ;
- prompt injection ;
- déploiement en échec.

Le laboratoire ne doit pas donner immédiatement la réponse complète.

---

## 8. Exercices

Chaque exercice doit comporter :

- une consigne précise ;
- des données d’entrée ;
- un résultat attendu ;
- un ou plusieurs indices ;
- une correction expliquée ;
- des critères de réussite ;
- des exemples acceptables ;
- des exemples insuffisants ;
- des critères bloquants lorsque nécessaire.

Les exercices doivent éviter les tâches purement déclaratives lorsque la notion peut être pratiquée.

---

## 9. Quiz

Les quiz doivent mesurer la compréhension, pas seulement la mémorisation.

Règles obligatoires :

- exactement cinq questions lorsqu’un quiz de leçon est requis ;
- bonnes réponses réparties entre les positions ;
- distracteurs crédibles ;
- scénarios réalistes ;
- aucune réponse absurde uniquement destinée à être éliminée ;
- explication de la bonne réponse ;
- explication de la raison pour laquelle les autres réponses sont moins adaptées.

Les quiz doivent varier :

- choix de conception ;
- diagnostic d’erreur ;
- lecture de trace ;
- compréhension de code ;
- sécurité ;
- ordre d’exécution ;
- identification d’un risque ;
- choix entre plusieurs solutions plausibles.

Un quiz seul ne valide jamais une compétence pratique.

---

## 10. Évaluation des compétences

Chaque module doit évaluer ce que l’apprenant sait réellement faire.

Une validation complète combine plusieurs preuves :

- quiz ;
- action réalisée ;
- livrable ;
- test automatique ;
- explication de l’apprenant ;
- preuve visible ;
- grille d’évaluation.

Une grille doit distinguer :

- insuffisant ;
- acceptable ;
- maîtrisé.

Elle doit préciser les critères bloquants.

La validation d’un module ne doit pas reposer uniquement sur une déclaration de l’apprenant.

---

## 11. Projets fil rouge

Chaque parcours doit utiliser un projet fil rouge lorsque cela est pertinent.

Le projet doit :

- évoluer module après module ;
- produire des éléments réutilisés ensuite ;
- représenter un cas d’usage crédible ;
- rester simple au début ;
- permettre des erreurs et corrections ;
- aboutir à une version publiable ou démontrable.

Le projet ne doit jamais être décoratif.

Chaque module doit préciser :

- l’état du projet avant le module ;
- la capacité ajoutée ;
- les fichiers ou artefacts produits ;
- la preuve de fonctionnement ;
- le principal risque ;
- la validation humaine nécessaire.

---

## 12. Carnet d’ingénieur

À la fin de chaque module, l’apprenant doit disposer d’une synthèse appelée « carnet d’ingénieur ».

Elle doit contenir :

- ce que l’apprenant sait désormais faire ;
- les notions à retenir ;
- les erreurs fréquentes ;
- les commandes, fichiers ou concepts importants ;
- ce que l’IA peut gérer ;
- ce que l’apprenant doit vérifier ;
- la décision de conception principale ;
- la prochaine étape.

Cette synthèse doit être concise et utile comme référence future.

---

## 13. Rapport de progression

À la fin d’un module, l’application doit pouvoir afficher :

- les compétences acquises ;
- les compétences en cours ;
- le livrable validé ;
- les quiz réussis ;
- les actions confirmées ;
- les tests ou évaluations passés ;
- les points à revoir.

La progression doit progressivement passer d’une logique de leçons terminées à une logique de compétences démontrées.

---

## 14. Rôle de l’IA dans l’apprentissage

L’IA est un assistant, pas un pilote automatique.

Elle peut :

- expliquer ;
- proposer ;
- générer un brouillon ;
- créer du code ;
- aider au diagnostic ;
- suggérer des tests ;
- relire un livrable.

L’apprenant doit obligatoirement :

- comprendre le rôle de la solution ;
- vérifier les entrées et sorties ;
- tester le résultat ;
- contrôler les risques ;
- valider les décisions importantes ;
- savoir expliquer ce qui a été construit.

Aucun parcours ne doit encourager le copier-coller aveugle.

---

## 15. Sécurité et responsabilité

La sécurité doit être intégrée tout au long des parcours.

Elle ne doit pas être isolée dans un seul module.

Selon le sujet, les contenus doivent couvrir progressivement :

- validation des entrées ;
- permissions limitées ;
- secrets ;
- données non fiables ;
- coûts ;
- délais ;
- conditions d’arrêt ;
- actions irréversibles ;
- supervision humaine ;
- journalisation ;
- récupération après erreur ;
- maintenance.

---

## 16. Standard de production des contenus

Avant publication, chaque module doit être vérifié selon les critères suivants :

- objectif mesurable ;
- une notion principale par leçon ;
- vocabulaire défini ;
- pratique réelle ;
- code expliqué ;
- erreurs réalistes ;
- quiz non prévisible ;
- exercice vérifiable ;
- livrable concret ;
- projet fil rouge réellement enrichi ;
- validation pratique ;
- sécurité intégrée ;
- ton accessible ;
- absence de surcharge inutile.

---

## 17. Règles pour les agents de développement

Avant de créer ou modifier un parcours, tout agent doit lire :

- `docs/AI_ACADEMY_VISION.md` ;
- `docs/AI_ACADEMY_LEARNING_STANDARD.md` ;
- le blueprint du parcours concerné ;
- les conventions de contenu existantes ;
- les tests liés au parcours.

L’agent doit :

- préserver la compatibilité avec l’existant ;
- éviter les nouveaux types de blocs non nécessaires ;
- ne pas réécrire un module sans justification ;
- signaler les hypothèses ;
- produire des tests ;
- vérifier les JSON ;
- lancer le lint et les tests ;
- expliquer les limites de l’environnement ;
- ne jamais fusionner automatiquement une Pull Request.

---

## 18. Critères de réussite d’un parcours AI Academy

Un parcours est réussi lorsque l’apprenant peut :

- expliquer les notions principales ;
- modifier une solution existante ;
- construire une version simple ;
- diagnostiquer une erreur ;
- tester son résultat ;
- justifier ses décisions ;
- utiliser l’IA sans dépendance aveugle ;
- publier ou démontrer un projet ;
- identifier les risques ;
- savoir ce qu’il doit encore apprendre.

---

## 19. Risques pédagogiques à éviter

- trop de théorie sans pratique ;
- trop de code sans explication ;
- leçons trop longues ;
- quiz prévisibles ;
- exercices sans preuve ;
- projets décoratifs ;
- sur-ingénierie ;
- dépendance excessive à un outil ;
- sécurité ajoutée trop tard ;
- progression uniquement basée sur la lecture ;
- sentiment de maîtrise sans capacité démontrée ;
- multiplication de fonctionnalités sans besoin pédagogique.

---

## 20. Principe directeur

Chaque contenu AI Academy doit répondre à cette question :

> À la fin, qu’est-ce que l’apprenant est réellement capable de faire qu’il ne savait pas faire avant ?

Si la réponse n’est pas claire, observable et vérifiable, le contenu doit être retravaillé.
